Strategy

The Strategy is a behavioural pattern that is used to define a family of algorithms that can be choosen at runtime.

This way, the client code can change the behavior of a given context at runtime, allowing for greater flexibility.

Motivation

Imagine that we are programming behavior for a house. Houses can have all kinds of doors - external doors, safe deposit doors, closet doors, sliding doors etc.

All these doors share common attributes and behaviors. For example, every door has some kind of locking and unlocking mechanism even if the exact functionality might differ.

If Door is our base class, we can just extend our base functionality in our specialized door classes. For example, we might choose to override the lock() and unlock() methods. This is not a bad strategy (no pun intended ^_^) to begin with. After all, by subclassing, we can reuse what we need and add new functionality as needed.

Now let's say that we decide that different doors can be opened different types of ways, which is realistic. For example, sliding doors will slide open and close, closet doors can also slide open and close. External doors can be revolving.

In addition, each door can be made out of a different material, and have different colors. By this point, you are probably able to see the problem with our current approach - it works when things are simple but as soon as we start customizing our doors, it falls apart.

The problem with this approach is that it overuses inheritance. If most subclasses have to override many of Door's methods, then we are not benefitting much from having a Door superclass. In other words, the class Door does not tell us much about doors because all of the doors have significantly different implementation. Some doors might be painted, some polished, some might have windows etc.

If we put all this in our Door class, we have now duplicated code across all classes. Even if we have a separate subclass for every type of Door, we are starting to lose the benefit of inheritance.

Not only this, but each time we want to add a new feature, a change in Door may affect all the classes in the inheritance hierarchy. If we add a method to Door then it affects all of its subclasses. This is referred to as tight coupling, meaning both the base classes and the parent classes rely heavily on each other's implementation. Good design adheres to loose coupling.

If that was not enough, we have now lost flexibility as well. Let's say that we wanted to create a custom safe deposit door, which locks using a keycard, made out of titanium, and has no windows. We would need to create a class called SafeDepositDoorKeyCardTitaniumNoWindows.

Because of this, the behavior of our given Door type is decided at compile time. We can't change this Door to be password protected at runtime. This flexibility is crucial as our application becomes more sophisticated.

Lastly, this design violates the Single Responsibility Principle (SRP) and onboarding new developers would also become challenging. Every time we need to add a new feature, developers would need to tread cautiously that making changes in one class does not break the entire application.

The solution

Solution 1: Leave all implementation for the subclasses

A solution might be to say that, well, just don't implement anything at Door and use interfaces instead. Particularly for the lock() and the unlock() methods, why not just have an interface called Lockable and have all the concrete classes implement that interface?

While not bad, the issue with this approach is that it allows no code reuse. Imagine having twenty or thirty different types of doors - each door will have to implement its own lock() and unlock() methods. Furthermore, to introduce a consistent change across all subclasses, we would have to modify every single one of them. We need a more balanced approach.

Solution 2: Encapsulate what varies

Our previous two candidate solutions, namely inheritance and interfaces did not provide us with the sophisticated design we are looking for.

Let's follow the design principle, "Encapsulate What Varies". This design principle tells us to identify parts of the applications that vary and separate them from what stays constant. We are altering our lock() and unlock() code every time we add a new type of Door. This way, we can implement and extend this behavior however we like, without affecting the rest of the application.

What varies?

  • Door opening and closing behavior: Different doors have their own ways to open and close.
  • Door locking and unlocking: Some doors have no locks, some unlock using a keycard, and others might ask you for a password.

All doors have a size and dimensions, so we can keep this in our Door class.

By delegating these varying behaviors using the Strategy pattern, our main Door class does not need to worry about the specifics. We can set our door to perform any locking or opening behavior we like.

The strategy pattern generally has the following key components:

1. Context (Navigator Class)

In our example this is the Door class.

Instead of determining its behavior directly or through a series of conditionals, it delegates this responsibility to the strategy it is provided. This allows the Context to change its behavior dynamically by swapping out its strategy.

2. Strategy Interface

The blueprint that ensures consistency across all strategies. To generalize, it contains some variation of an execute() method. In our case, we may have an Openable interface which defines the lock() and unlock() methods.

3. Concrete Strategies

These classes implement the strategy interface, defining the specific behavior or algorithm. The client will choose which concrete strategy to use and pass it to the context.

4. Client

The Client interacts with the Context and decides which Concrete Strategy to use. It's the Client's responsibility to create strategy objects and pass them to the Context. By doing so, the Client can change the strategy dynamically based on certain conditions or preferences without needing to modify the Context or Strategy classes themselves.

This can also be referred to as the Driver Class which instantiates the different strategies. The client can add/remove features on the fly to the Context. In our case, the client can make the Door open a different way and lock a different way, all at runtime.

Design and Implementation

from abc import ABC, abstractmethod

class Lockable(ABC):
    @abstractmethod
    def lock(self):
        pass

    @abstractmethod
    def unlock(self):
        pass

class NonLocking(Lockable):
    def lock(self):
        print("Door does not lock - ignoring")

    def unlock(self):
        print("Door cannot unlock because it cannot lock")

class Password(Lockable):
    def lock(self):
        print("Door locked using password!")

    def unlock(self):
        print("Door unlocked using password!")

class KeyCard(Lockable):
    def lock(self):
        pass

    def unlock(self):
        pass

class Openable(ABC):
    @abstractmethod
    def open(self):
        pass

    @abstractmethod
    def close(self):
        pass

class Standard(Openable):
    def open(self):
        print("Pushing door open")

    def close(self):
        print("Pulling door closed")

class Revolving(Openable):
    def open(self):
        pass

    def close(self):
        pass

class Sliding(Openable):
    def open(self):
        pass

    def close(self):
        pass

class Door(ABC):
    def __init__(self):
        self._lock_behavior = None
        self._open_behavior = None

    def set_lock_behavior(self, lock_behavior):
        self._lock_behavior = lock_behavior

    def set_open_behavior(self, open_behavior):
        self._open_behavior = open_behavior

    def perform_lock(self):
        self._lock_behavior.lock()

    def perform_unlock(self):
        self._lock_behavior.unlock()

    def perform_open(self):
        self._open_behavior.open()

    def perform_close(self):
        self._open_behavior.close()

    def get_dimensions(self):
        return 5

class ClosetDoor(Door):
    pass

class ExternalDoor(Door):
    pass

class SafeDepositDoor(Door):
    pass

class SlidingDoor(Door):
    pass

# Initialize door
c = ClosetDoor()

# Set behaviors
c.set_open_behavior(Standard())
c.set_lock_behavior(NonLocking())

# Invoke behaviors
c.perform_open()g
c.perform_close()
c.perform_lock()
c.perform_unlock()

# Upgrade the door to be password protected
c.set_lock_behavior(Password())
c.perform_lock()
c.perform_unlock()

A common alternative to the strategy pattern is to simply pass in an inline / lambda function, which allows us to extend the behavior of a method or class. Our implementation above is the classic strategy, though a quick google search should show you how to implement using lambda as well.

Limitations and Pitfalls

It is important for us to remember that the strategy pattern is not a replacement for inheritance, meaning it shouldn't be used just because you don't want to use inheritance. When a class has a behavior that can be realized in multiple ways, the use of the strategy pattern is a good candidate solution.

Perhaps the two most important limitations and pitfalls of the pattern are:

1. Increased Number of Classes

Every new strategy usually requires a new interface, which means it requires a new concrete strategy class.

2. Risk of Over-Engineering

The strategy pattern provides great flexibility when there is varying behavior. This can be a double-edged sword. It's tempting to use the strategy pattern wherever there's a hint of varying behavior, but sometimes a simpler solution like conditional logic might be more appropriate. Implementing the strategy pattern in places where it's not truly needed can make the design unnecessarily complex.

Classic Use Cases

1. Payment Processing Systems - any system where users can choose to pay with credit card, PayPal, Bitcoin etc.

2. Travel Routes - We can customize our routes when traveling. For example, we might choose a route without highways, the shortest route, or the fastest route. Each of these routes have different strategies.

3. Shipping Options - Each shipping method has its own category and its own strategy for determining cost and delivery time.

Closing Notes

The entire idea of using design patterns is to ensure sophisticated and flexible design - these are tried and tested methods that will always work, given that the scenario is right. With the strategy pattern, we have effectively achieved the following:

  1. Separation of Concerns: We have separated out behaviors (Sliding, Password, etc.) from the Door class. Each behavior encapsulates a specific way of opening, closing, locking, or unlocking.

  2. Interchangeable Strategies: We can swap out behaviors at runtime because we set the opening behavior and the locking behavior in our main class. Again, the Door class remains completely untouched.

  3. Composition over inheritance: As we saw in the UML, instead of using inheritance to define every single behavior, our Door class has a composition association with both the Lockable and Openable interfaces. This allows us to have much greater flexibility.

It might help to think of composition as an "HAS-A" relationship and inheritance as an "IS-A" relationship. It is easier to change something you have than something you are.
