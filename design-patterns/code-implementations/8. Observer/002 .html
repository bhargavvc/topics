Observer

The observer is a behavioural pattern to send notifications through our code which are triggered by certain state changes.

Motivation

Let's say we run an online bookstore that sells books. From a software perspective, the challenge we face is designing a system so that customers can be notified when books of their interest comes back in stock. A naive solutions would be for the customer to constantly check the website to see if the book is back in stock - in programming terms, this is called polling.

1. Polling

The system maintains information about product availability, and customers must initiate the request to retrieve this information.

One of the issues with this approach is efficiency. The constant polling by customers will lead to an increased load on the system, and it also increases customer effort. This means that customers have to actively engage in checking the product status, which isn't an optimal user experience. Furthermore, there will be many instances where customers end up polling when there is no actual update.

Polling is a popular choice, especially in systems where immediate notifications are not a priority. Such systems might want to avoid the complexity of managing and sending notifications to a large number of customers. For our system, we want to be notified as soon as the desired book is back in stock.

So, what do we do? Well, now we understand that our ideal solution should be loosely coupled, use Inversion of Control (IoC), and adhere to the Single Responsibility Principle. This is where the observer pattern comes into play.

The solution

Loose coupling is a design principle wherein the component or the classes within a system are designed to have as little dependency on each other as possible. So while the components are still connected (or coupled) they don't know too much about each other or rely too much on each other, which makes them loosely coupled. The components in the observer pattern interact with each other through well-defined interfaces, rather than concrete implementations, enabling interchangeability and adaptability.

When a customer is interested in a specific book and it's out of stock, the customer(s) subscribe(s) to receive notifications for that book, becoming observers of its stock status. The bookstore or the inventory system acts as the subject (or publisher), responsible for sending out the notifications that the customers have requested. It maintains a list of the observers who want to be informed about stock changes for specific books. When the stock level of a book changes, the subject sends out a notification that the observers receive. Any observer can add or remove itself from the subscription list at any time.

1. Subscription - Customers subscribe to receive notifications for specific books by providing their email.

2. Stock Change - When the stock level of a book changes, the bookstore's inventory system automatically sends notifications to all subscribed customers.

3. Notification - Customers receive notifications about the change in stock level and can take appropriate action based on this information.

The technique above employs inversion of control (IoC). Think about it like this: instead of A calling B, A signs up for a callback from B. B takes the communication initiative instead of A. This is useful when the calling depends on the state of B.

Design and Implementation

The observer pattern typically contains of the following components:

1. Subject (Publisher) - This interface/abstract class specifies the methods that our concrete subject must implement. It always contains these three methods, though is not limited to them.

  • registerObserver - allows the observers to subscribe to event changes.
  • removeObserver - allows the observers to unsubscribe to event changes.
  • notifyObservers - makes sure all the observers are notified once the data in the subject changes.

2. Concrete Subject - This will implement the Subject interface. The ConcreteSubject manages a list of observers, and will manage the data that the observers are interested in observing the change for. If the state changes, it can decide to notify all registered observers using the notifyObservers method.

3. Observer (Interface/Abstract class) - This interface/abstract class declares an update method which is called to inform the observer of changes in the subject.

4. ConcreteObserver - This is the concrete class implementation which implements the Observer and contains a reference to the Subject via composition. Upon instantiation, it often registers itself with the given subject, and it starts getting updates. It implements the update method and this method will allow the Observer to handle the notification.

There are technically two implementations of the observer pattern we can implement. There is a push model and a pull model. The push model directly updates all the observers after a change in value of the subjects. In the pull model, the publisher only informs the subscribers that there has been a change and the subscribers are responsible for fetching the books themselves if they are interested. In our example, we use the push model.

The UML diagram and the code for the above will look like the follows:

The UML template above can be applied to any scenario which uses the observer pattern.

import java.util.ArrayList;

abstract class Subject {

  abstract void registerObserver(Observer o);

  abstract void removeObserver(Observer o);

  abstract void notifyObservers();
}

class ConcreteSubject extends Subject {

  private ArrayList<Observer> observers = new ArrayList<Observer>();
  private int value = 0;

  @Override
  public void registerObserver(Observer o) {
    observers.add(o);
  }

  @Override
  public void removeObserver(Observer o) {
    observers.remove(o);
  }

  @Override
  public void notifyObservers() {
    for (Observer observer : observers) {
      observer.update(value);
    }
  }

  public void setValue(int value) {
    this.value = value;
    notifyObservers();
  }
}

interface Observer {
  public void update(int value);
}

class ConcreteObserver implements Observer {

  private int value;
  private Subject subject;

  public ConcreteObserver(Subject sSub) {
    subject = sSub;
    subject.registerObserver(this);
  }

  @Override
  public void update(int value) {
    this.value = value;
  }
}

Using the simple template above, we can apply it to our bookstore example. In our example, the subject is the bookstore and the observers are the customers who have subscribed to be notified when there is a change in the stock level of the books. When a book comes back in stock, we will update the inventory and notify all of the customers.

Following the exact same UML and the code template, it is simple to apply to to our scenario:

UML class diagram of our observer pattern applied to the bookstore example.

interface Store {
  void addCustomer(Customer c);

  void removeCustomer(Customer c);

  void notifyCustomers();
}

class BookStore implements Store {

  private List<Customer> customers = new ArrayList<Customer>();
  private int stockQuantity = 0;

  @Override
  public void addCustomer(Customer c) {
    customers.add(c);
  }

  @Override
  public void removeCustomer(Customer c) {
    customers.remove(c);
  }

  @Override
  public void notifyCustomers() {
    for (Customer customer : customers) {
      customer.update(stockQuantity);
    }
  }

  public void updateQuantity(int quantity) {
    this.stockQuantity = quantity;
    notifyCustomers();
  }
}

interface Customer {
  public void update(int stockQuantity);
}

class BookCustomer implements Customer {

  private int observedStockQuantity;
  private Store store;

  public BookCustomer(Store store) {
    this.store = store;
    store.addCustomer(this);
  }

  @Override
  public void update(int stockQuantity) {
    this.observedStockQuantity = stockQuantity;
    if (stockQuantity > 0) {
      System.out.println(
        "Hello, A book you are interested in is back in stock!"
      );
    }
  }
}

class Main {

  public static void main(String[] args) {
    Store store = new BookStore();

    // Create email notifiers (observers) for two customers
    Customer customer1 = new BookCustomer(store);
    Customer customer2 = new BookCustomer(store);

    // Initially, the book is out of stock
    System.out.println("Setting stock to 0.");
    store.updateQuantity(0);
    // The book comes back in stock
    System.out.println("\nSetting stock to 5.");
    store.updateQuantity(5);

    // Remove customer1 from the notification list
    store.removeCustomer(customer1);

    // Simulate the situation where the stock changes again
    System.out.println("\nSetting stock to 2.");
    store.updateQuantity(2);
  }
}

Limitations and Pitfalls

1. Processing Overhead - If there are many observers or if the notify operations are computationally expensive, it can lead to performance issues. Every time a change occurs in the subject, all its observers need to be updated. This can cause a cascading effect where one change leads to a series of updates, which can significantly slow down the system.

2. Unintended side effects - Firstly, the order in which the observers are notified is not explicitly managed. This means that different notification orders can actually lead to different states, which might become a source of bugs and will require effort to track down. Secondly, as the number of observers grows, debugging and understanding the flow of the program become more challenging.

Classic Use Cases

1. Home Automation Systems - In a smart home, various devices can act as observers. For instance, a temperature sensor might monitor room temperature. When the temperature crosses a certain threshold, the air conditioning system, an observer, gets activated automatically.

2. Gaming - Achievement Systems - In many video games, players can earn achievements or trophies for completing specific tasks or challenges. The game can use the Observer pattern to monitor various game metrics or player actions. When a particular metric meets the criteria for an achievement, the corresponding observer triggers, granting the player the achievement.

3. Subscription Services - Systems where users can subscribe to receive news, updates, or other information. When new content is available, all subscribed users (observers) receive a notification, such as an email or a push notification.

4. Real-Time Data Monitoring - Applications that monitor real-time data, such as stock market applications, can use the Observer pattern. When the stock price changes, all subscribed clients (observers) are immediately notified and can update their displays or perform other actions.

Closing Notes

Given the observer pattern, we effectively achieve the following:

1. Loose coupling - The Subjects and Observers are loosely coupled; they interact but have little knowledge of each other. All the Subject knows is that the Observer implements the Observer interface. Any class may implement the Observer interface and participate in subscribing to changes pushed out by the Subject. If we want to add a new type of Observer, we never have to modify the Subject.

2. Inversion of Control (IoC) - The Observer pattern flips the control mechanism between objects. Rather than one object actively checking another for changes (polling), control is inverted: the observed object notifies its registered observers when there's a change. Both the Subject and the Observers rely on common abstract interfaces (the Subject and Observer interfaces), emphasizing dependency on abstractions over concrete classes.
