Facade

A Facade is an outward appearance that is maintained to conceal reality.

In programming, though, the Facade is a structural pattern that presents a complicated collection of classes in a simplified manner. This allows clients easy access to the functionality of these classes.

We use the facade design pattern when we:

  • Desire to encapsulate and abstract the underlying implementation details of classes.
  • Desire a single access point to a module, layer, or library.

Motivation

Imagine we've just moved into a smart home equipped with a smart lighting system, climate control, security, and an entertainment system. Now, let's say we want to set a particular mood, such as "movie."

From a software perspective, the challenge is ensuring the client's interaction with the subsystem is seamless. Transitioning to "movie" would require multiple adjustments: setting the lights, adjusting the temperature, configuring the security, and so on. Additionally, we must consider which settings we want the client to access.

The straightforward, albeit, naive solution would provide the client with access to every single setting, requiring them to manage these functionalities manually. This means the client would need:

  • Smart Lighting setting - to adjust the lights.
  • Climate Control setting - to set the desired temperature.
  • Security System setting - to arm or disarm the system.
  • Entertainment System setting - to control the TV and streaming services.

This setup is clearly impractical. The necessity to navigate through four separate apps or settings detracts from the convenience and intuitiveness of a "smart" home. Moreover, there might be a specific sequence in which these adjustments should be made. Such decisions shouldn't be left to the client.

This ties in beautifully with the Open/Closed Principle. If these decisions are left to the client, the client will need to make modifications if there is an update in the SmartHome.

To illustrate this in code, consider the following example. In the Client class, you'll observe that the Client has full access to all methods within the SmartHome class, exposing the detailed intricacies of the class and burdening the client with complexities.

class SmartHome {
    constructor() {
        this.lights = Brightness.UNKNOWN;
        this.temperature = 19;
        this.isSecurityArmed = false;
        this.stream = Service.UNKNOWN;
    }

    setBrightness(b) {
        this.lights = b;
    }

    setTemperature(temp) {
        this.temperature = temp;
    }

    setIsSecurityArmed(securityStatus) {
        this.isSecurityArmed = securityStatus;
    }

    setStreamingService(s) {
        this.stream = s;
    }
}

// Client simulation
const setup = new SmartHome();
setup.setBrightness(Brightness.DIM);
setup.setTemperature(21);
setup.setIsSecurityArmed(false);
setup.setStreamingService(Service.NETFLIX);

As the solution stands right now, the client has to configure every single subsystem individually. Now consider a more complex sequence or interdependencies among systems and you quickly realize the impracticality. The user experience will deteriorate rapidly. However, most importantly, this approach breaches the principle of Encapsulation - which is one of the four pillars we discussed in the introduction.

The client is so deeply entrenched in the smart home's internal operations that it compromises modularity. How so? Well, any change to SmartHome cascades to the client.

Also recall our discussion on the interface segregation principle from the adapter article. Providing the client with access to every SmartHome method, even the irrelevant ones, forces unnecessary dependencies. It should be on a need to know basis.

This is where the Facade pattern would shine, offering a more streamlined interface for common tasks or scenarios.

The solution

The bottom-line with the facade pattern is: it provides a simplified interface to a more complex subsystem, hiding the intricate details, making it easier for the clients to use.

Instead of the client being bogged down in all kinds of details, the facade encapsulates subsystem's functionality and the client can treat all of the intricate details as a black box. This way, we not only keep the code cleaner but also add a layer of abstraction between the client and the subsystem, making the system loosely coupled.

Most importantly, we have now unified all of the processes into a single operation.

Design and Implementation

Try to think of the Facade pattern as the 'concierge' of software design. Instead of guests at a hotel interacting with every department individuallyhousekeeping, room service, front desk, bell servicethey speak to the concierge, who orchestrates everything behind the scenes.
The guests enjoy a seamless experience, unaware of the intricate coordination happening in the background. That's the power and simplicity that facade brings to software design.

The facade pattern typically consists of the following components:

1. Subsystem classes - These are the classes the facade aims to simplify. Each subsystem class encapsulates specific functionalities, such as lighting or temperature control. They house the intricate details and operations, like setBrightness() and setTemperature(), that the facade will abstract away from direct client access.

2. Facade - The Facade serves as a gateway, offering a unified, higher-level interface to the diverse interfaces within a subsystem. By encapsulating the complexities of the subsystem's operations, it simplifies client interaction.

3. Client - The client interacts with the subsystem primarily through the Facade, gaining a streamlined and intuitive experience. Thanks to the facade's abstractions, clients can focus on their tasks without getting entangled in the subsystem's complexities. However, it's important to note that if necessary, clients can still retain the option to bypass the facade and access the subsystem directly.

    setBrightness(brightness) {
        this.brightness = brightness;
    }

    setTemperature(temperature) {
        this.temperature = temperature;
    }

    setIsSecurityArmed(isSecurityArmed) {
        this.isSecurityArmed = isSecurityArmed;
    }

    setStreamingService(streamingService) {
        this.streamingService = streamingService;
    }

    enableMotionSensors() {
        // Implementation here
    }

    updateFirmware() {

    }

class SmartHomeFacade {
    constructor(smartHome) {
        this.smartHome = smartHome;
    }

    setMovieMode() {
        this.smartHome.setBrightness(Brightness.DIM);
        this.smartHome.setTemperature(21);
        this.smartHome.setIsSecurityArmed(false);
        this.smartHome.setStreamingService(Service.NETFLIX);
    }

    setFocusMode() {
        this.smartHome.setBrightness(Brightness.BRIGHT);
        this.smartHome.setTemperature(22);
        this.smartHome.setIsSecurityArmed(true);
        this.smartHome.setStreamingService(Service.UNKNOWN);
    }
}

const facade = new SmartHomeFacade(new SmartHomeSubSystem());
facade.setMovieMode();
facade.setFocusMode();

Limitations and Pitfalls

Let's take a look at two limitations of this pattern:

1. Over-Simplification - The primary function of the facade is to simplify the access to the subsystem, but it can sometimes lead to over-simplification. If the facade does not provide enough functionality for specific use cases, the clients will need to bypass the facade (which defeats the purpose because it exposes the system's complexity). This now calls for increased maintenance work where the system may need to be refactored to accommodate these advanced use cases.

2. Maintenance Challenges - When changes occur in the underlying subsystem, the Facade might need regular updates. This can make its maintenance challenging. If hundreds of clients are relying on the Facade and it's not updated appropriately, the ripple effect of those changes can be significant.

Classic Use Cases

1. API Wrappers - Many times, third-party libraries provide APIs that are very granular or complex. For example, X (formerly Twitter) provides a comprehensive API but it can be cumbersome, especially for those unfamiliar with intricacies of OAuth, endpoints structures. Tweepy is a python library, which acts as a facade, simplifying authentication and the process of sending tweets.

2. GUI Libraries - Complex GUI libraries can have intricate setups for creating windows, shapes etc. A facade might be used to allow the developers to make simplified calls to abstract these details, allowing developers to create UI elements with simple method calls.

3. Database Access using ORM - Instead of writing raw SQL queries, developers can use high-level methods to retrieve, insert or update records. For example, a piece of code like user = DB.get_record('users', 5) abstracts away the details of forming the SQL query, executing the query, fetching the result and handling potential errors.

Closing Notes

You might have noticed that the facade was easier to understand than the rest of the design patterns. It's important to note that design patterns don't always have to be complex and difficult to motivate. From a design perspective, the facade pattern achieves the following:

1. Abstraction of Complexity - The facade pattern simplifies a complex subsystem by providing a unified, high-level interface. It promotes ease of interaction and reduces the cognitive load on the client class.

2. Decoupling - Facade offers modularity and decouples the subsystem from the client, creating a more modular architecture. As long as the facade's interface remains consistent, changes in the subsystem's implementations will not directly affect clients.
